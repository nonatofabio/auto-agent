## Context

The orchestrator already supports creating jobs (`create-job`) and running a baseline evaluation (`run-baseline-evals`). After the baseline is established, the developer needs a way to trigger the autonomous optimization loop that iteratively improves the target agent.

This spec covers `npm run run-job -- --id <job-id>` — the main loop script. It reads the baseline results, then enters a loop where each iteration spawns Claude Code to analyze failures, hypothesize a fix, implement it, run evals, and report whether to keep or discard the changes. The orchestrator (this script) handles all branching, rollback, and loop control. Claude Code handles all analysis, coding, and evaluation within a single invocation per iteration.

---

## Non-Goals

- Multi-metric optimization (Pareto frontiers, secondary constraint thresholds) — single metric (accuracy) for now
- Parallel hypothesis execution — strictly sequential
- Constraint validation (forbidden files check via `git diff`) — defer to a follow-up
- Memory Manager as a separate summarizer agent — Claude Code updates MEMORY.md directly during its run
- Final job report (`final-REPORT.md`) generation — defer to a follow-up
- Web dashboard or status UI

## Acceptance Criteria

- Running `npm run run-job -- --id <job-id>` starts the optimization loop for an existing job
- The script exits with an error if the job folder doesn't exist
- If the baseline hypothesis (`000-baseline`) doesn't exist, the script runs the baseline evaluation automatically before entering the loop
- Each iteration creates a hypothesis folder at `jobs/<job-id>/hypotheses/<id>/` where `<id>` is a random 6-character alphanumeric string (e.g., `a3f9b2`)
- Each iteration copies `REPORT-TEMPLATE.md` into the hypothesis folder as `REPORT.md` before spawning Claude Code
- Each iteration creates a git branch in the target repo named `<job-id>-hyp-<id>` from the current branch
- Claude Code is spawned with a system prompt that injects: the JOB.md contents, the MEMORY.md contents, the baseline REPORT.md contents (`000-baseline/REPORT.md`), the hypothesis ID, the REPORT.md path, and instructions to analyze failures, implement a fix, run evals, and fill in the report
- The system prompt explicitly instructs Claude Code to update `MEMORY.md` before finishing — recording what was tried, what worked/failed, and any patterns observed. This is mandatory, not optional.
- Claude Code fills in all sections of `REPORT.md`, including the Recommendation section with exactly `**Decision: CONTINUE**` or `**Decision: ROLLBACK**`
- After Claude Code exits, the orchestrator parses the Recommendation from `REPORT.md`
- If CONTINUE: the loop records the current branch as the new "best" branch and proceeds to the next iteration from it
- If ROLLBACK: the orchestrator checks out the previous "best" branch (discarding the hypothesis branch) before the next iteration
- The script accepts `--max-iterations <n>` as an optional CLI argument (default: 5). The loop stops after this many iterations.
- CLI usage: `npm run run-job -- --id <job-id> --max-iterations 3`
- Each hypothesis is recorded in the SQLite `hypotheses` table (via existing `createHypothesis` util) with status updated to `accepted` or `rejected` after the decision
- The script streams Claude Code's NDJSON output to stdout (same pattern as `run-baseline-evals.ts`)
- The script prints a summary after each iteration: hypothesis ID, decision, and current accuracy

- Reuse existing utilities: `createHypothesis` from `utils/create-hypothesis.ts`
- The system prompt for Claude Code must be generated using the prompt-engineering skill (EARS syntax + VERIFY checklist)

## Technical Notes

- The baseline script (`run-baseline-evals.ts`) is the reference implementation for spawning Claude Code, streaming NDJSON, parsing JOB.md, and git operations. The new script should follow the same patterns.
- The `createHypothesis` util is idempotent — it won't fail if re-run for an existing hypothesis.
- The SQLite schema already has `hypotheses` (id, statement, status, branch_name, created_at, completed_at, reasoning) and `eval_runs` / `eval_cases` tables. The hypothesis `statement` field should be filled in by Claude Code as part of the report.
- Random 6-char alphanumeric ID: `crypto.randomBytes(3).toString("hex")` gives exactly 6 hex chars.
- The Recommendation parsing should look for `**Decision: CONTINUE**` or `**Decision: ROLLBACK**` in the REPORT.md content. If neither is found, treat as ROLLBACK (safe default).
- The "previous best branch" starts as the baseline branch (`<job-id>-baseline`) and advances each time a CONTINUE decision is made.
- MEMORY.md is given to Claude Code via `--add-dir` (the job dir is already added). Claude Code reads and updates it directly during its run. The system prompt must make updating MEMORY.md a mandatory step in the VERIFY checklist.
- The baseline REPORT.md (`jobs/<job-id>/hypotheses/000-baseline/REPORT.md`) is read by the orchestrator and injected into the system prompt as context. This gives Claude Code visibility into the baseline metrics and failing cases without needing to re-run evals to understand the starting state.

## Decision Log

| Decision | Rationale |
|----------|-----------|
| Single Claude Code invocation per iteration (analysis + implementation + eval + report) | Simpler architecture, fewer failure modes. Claude Code can read the baseline report, MEMORY.md, and codebase all in one session. Splitting into separate analyzer/implementer/evaluator agents adds coordination complexity with no clear benefit for v1. |
| Random 6-char hex ID instead of sequential numbering | Avoids collisions if hypotheses are created concurrently in the future. Sequential IDs require reading existing hypotheses to determine the next number. |
| ROLLBACK as safe default when recommendation can't be parsed | If Claude Code crashes or produces a malformed report, we shouldn't advance to a potentially broken state. |
| MEMORY.md updated by the same Claude Code instance (no separate summarizer) | Keeps the loop simple. Claude Code already has full context about what it tried and what happened. A separate summarizer agent can be added later if memory quality degrades. |
